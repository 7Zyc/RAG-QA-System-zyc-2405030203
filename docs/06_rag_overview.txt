RAG（Retrieval-Augmented Generation，检索增强生成）是一种将"信息检索"
与"文本生成"相结合的技术范式。它在生成答案前先从外部知识库中检索相关内容，
再将这些内容作为上下文交给大模型，从而显著提升答案的准确性与时效性。

一、RAG 的动机
- 大语言模型（LLM）存在"幻觉"：可能编造看似合理但实际错误的内容。
- LLM 的知识存在"截止时间"，无法回答最新问题。
- 私有数据无法被通用 LLM 直接访问。
- 重新训练或微调 LLM 成本高昂。

二、RAG 的工作流程
1. 索引阶段（Indexing）
   - 文档加载：PDF / DOCX / HTML / 数据库等。
   - 文本分块：使用 RecursiveCharacterTextSplitter 等切分器。
   - 向量化：使用 Embedding 模型（如 bge、nomic-embed-text）将文本块转为向量。
   - 存储：写入向量数据库（Chroma、FAISS、Milvus、Weaviate、Qdrant 等）。

2. 检索阶段（Retrieval）
   - 用户问题经相同 Embedding 模型转为 Query 向量。
   - 在向量库中做相似度检索（Top-K），返回若干相关文本块。
   - 可选：使用 BM25、HyDE、Re-rank 等方法进一步提升质量。

3. 生成阶段（Generation）
   - 将检索到的文本块 + 对话历史 + 用户问题拼接为 Prompt。
   - 调用 LLM 生成最终答案。
   - 系统提示词中通常要求"只依据提供的文档回答"。

三、关键参数
- chunk_size：文本块大小，过小则信息不完整，过大则检索噪声大。
- chunk_overlap：相邻块重叠长度，保留上下文衔接。
- top_k：检索返回的块数，一般 3-10。
- 嵌入模型选择：影响检索质量，bge-m3、nomic-embed-text 等较常用。
- Re-rank：使用 bge-reranker 等模型二次排序。

四、典型应用
- 企业知识库问答（规章制度、产品手册）。
- 客服机器人（FAQ + 业务文档）。
- 学术论文助手（个人文献库）。
- 法律、医疗、金融等垂直领域问答。
- 私域数据对话（笔记、邮件、聊天记录）。

五、优势与局限
优势：
- 显著减少幻觉，提升答案可追溯性。
- 知识可实时更新，无需重训模型。
- 私有数据不出本地，安全性高。
局限：
- 检索质量受限于嵌入模型与分块策略。
- 长文档、多跳推理仍有挑战。
- 对向量库的维护与版本管理有要求。

六、本地化 RAG 实践栈（本项目）
- Ollama：在本地运行 DeepSeek-R1、Qwen2 等开源大模型。
- LangChain：编排文档加载、分块、检索、生成流程。
- Chroma：轻量级本地向量数据库。
- Streamlit：构建交互式 Web 界面。
- PyInstaller：将 Web 应用打包为 Windows 可执行文件。

RAG 是当前企业级大模型应用最主流的"性价比"方案之一。
